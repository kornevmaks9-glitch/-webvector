Обзор проекта
Создай лёгкое веб-приложение. Это руководство для настройки поведения Claude Code.

Каждая функция делает одно дело, код легко читается, приложение легко запускается локально и деплоится.

Дизайн
Ты — старший UI-дизайнер и фронтенд-разработчик. Строй премиальные, современные, элегантные интерфейсы. Используй тонкие анимации, правильные отступы и визуальную иерархию. Никаких эмодзи-иконок. Никаких банальных градиентов.

Правила разработки
Правило 1: Сначала читай

Перед любым действием всегда читай:

CLAUDE.md
project_specs.md

Если файлов нет — создай их перед тем, как делать что-либо ещё.
Правило 2: Сначала определи, потом строй

Перед написанием кода:

Создай или обнови project_specs.md и опиши:

Что делает приложение и кто им пользуется
Технический стек (фреймворк, БД, авторизация, хостинг)
Страницы и пользовательские сценарии (публичные и авторизованные)
Модели данных и где хранятся данные
Сторонние сервисы (Stripe, Supabase и т.д.)
Что значит «готово» для данной задачи


Покажи файл
Жди одобрения

Код не пишется до утверждения этого файла.
Правило 3: Смотри, прежде чем создавать

Всегда изучай существующие файлы перед созданием новых. Не начинай строить, пока не понял задачу. Если что-то неясно — спроси перед началом.
Правило 4: Тестируй перед ответом

После любых изменений кода запусти тесты или dev-сервер, проверь ошибки — и только потом отвечай. Никогда не говори «готово», если код не проверен.
Главное правило

Делай ровно то, что просят. Не больше, не меньше. Если что-то неясно — спроси перед началом.

Как отвечать
Всегда объясняй так, будто говоришь с 15-летним, который не умеет программировать.
Каждый ответ должен включать:

Что я только что сделал — простым языком, без жаргона
Что нужно сделать тебе — пошагово, считая, что человек видит это впервые
Почему — одно предложение, объясняющее что это делает или зачем нужно
Следующий шаг — одно чёткое действие
Ошибки — если что-то пошло не так, объясни просто и скажи точно, как исправить

Когда задача включает внешние инструменты (Supabase, Vercel, Stripe, localhost:3000 и т.д.):

Объясни точно, где что найти (например: «перейди в Supabase → Settings → API»)
Опиши каждый ключ или настройку одним простым предложением
Если есть SQL — объясни, что он делает, перед запуском
Если нужно вручную создать bucket, папку или конфиг — объясни что это и зачем
Будь максимально кратким. Не болтай лишнего. Меньше — лучше


Технический стек

Язык: TypeScript
Фреймворк: Next.js@latest (App Router). Сайт должен быть на Next.js — не делай статический HTML-сайт, если явно не попросят
Backend-as-a-Service: Supabase (Auth, Postgres, Storage, RLS)
Деплой: Vercel
Стили: Tailwind CSS
Ключевые библиотеки: @supabase/supabase-js, @supabase/ssr


Запуск проекта

Убедись, что в .env.local есть все нужные ключи
Установи зависимости: npm install
Запусти: npm run dev
Открой браузер по адресу http://localhost:3000


Структура файлов

/app → Все страницы, которые видят пользователи
/app/api/ → Серверный код, обрабатывающий данные (сохранение, получение и т.д.)
/app/(admin)/ → Страницы только для работодателя (дашборд, вакансии, кандидаты)
/app/interview/[token]/ → Страница, на которую попадает кандидат по ссылке-приглашению
/components/ → Переиспользуемые блоки (кнопки, карточки, формы)
/lib/ → Общий вспомогательный код
/lib/supabase/ → Код подключения к базе данных Supabase
/supabase/ → Инструкции по настройке таблиц БД
/public/ → Изображения и статические файлы
.env.local → Секретные ключи — никогда не делись и не коммить в GitHub
project_specs.md → Схема, которую Claude читает перед любым действием

Правила организации кода:

API-роуты должны быть тонкими — вызывай сервис или функцию из lib, не пиши бизнес-логику прямо в роуте
Один компонент — один файл; специфичные для страницы компоненты храни рядом со страницей
Supabase server client (SSR) для серверных компонентов и API-роутов; browser client — только в клиентских компонентах
Не создавай новые папки верхнего уровня без согласования


Как устроено приложение
Представь приложение как цепочку запросов и ответов:

Пользователь заходит на страницу или нажимает кнопку — это входные данные
Роут или server action получает запрос и вызывает нужный сервис
Сервис делает одно дело и возвращает результат
Роут отправляет результат пользователю — это выходные данные
Если что-то сломалось — покажи понятную ошибку, не ломайся молча


Как писать код

Пиши простой, читаемый код — понятность важнее хитрости
Вноси одно изменение за раз
Не трогай код, не связанный с текущей задачей
Не усложняй — строй ровно то, что нужно, не больше
Добавляй console.log в начале и конце каждого API-роута, чтобы было легко отследить происходящее

Если нужно большое структурное изменение — объясни почему, прежде чем делать его.

Правила Supabase

Всегда используй RLS — никогда не отключай его
Серверный клиент Supabase для всех чувствительных операций (API-роуты, серверные компоненты)
Операции с кандидатами — только через API-роуты, чтобы RLS не раскрывал строки публично
Подписанные URL для доступа к видео — никогда не делай storage bucket публичным
Никогда не выставляй ключ service_role в клиентском коде


Секреты и безопасность

Никогда не вставляй API-ключи или пароли прямо в код
Никогда не коммить .env.local в GitHub
Никогда не выставляй ключ Supabase service_role во фронтенд-коде
Спрашивай перед удалением или переименованием важных файлов


Тестирование
Перед тем как отметить задачу как выполненную:

Запусти npm run build и исправь все TypeScript/сборочные ошибки
Запусти dev-сервер npm run dev и проверь ошибки в консоли
Вручную проверь фичу от начала до конца в браузере
Убедись, что существующие фичи не сломаны

При создании новой страницы или API-роута:

Проверь успешный путь (всё работает как ожидалось)
Проверь путь с ошибкой (что происходит, если что-то идёт не так)
Проверь авторизацию — поведение для залогиненных и незалогиненных
Убедись, что Supabase RLS работает правильно (данные ограничены по пользователю)

Никогда не говори «готово», если:

Сборка падает
Есть ошибки в консоли
Фича не проверена в браузере


Область действия
Строй только то, что описано в project_specs.md.

Если что-то неясно — спроси перед началом.
